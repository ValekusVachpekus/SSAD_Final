# Composite (Компоновщик)
**Категория:** Structural (Структурный)

## 📌 Суть (Intent)
Паттерн **Composite** позволяет сгруппировать объекты в **древовидные структуры** и работать с этими структурами так, как если бы они были единичным объектом.

## 🧠 Фундаментальная идея (из Лекции 10)
На лекции приводилась "бриллиантовая проблема": как единообразно работать с атомарными (Terminal/Leaf) и сложными (Composite) объектами?
*Примеры из лекции:* 
* Графика: Линия/Точка (Leaf) и Картинка (Composite, состоящий из линий).
* Файловая система: Файл (Leaf) и Папка (Composite, состоящая из файлов и других папок).
* Abstract Syntax Tree (AST) компилятора: переменные (Leaf) и логические выражения (Composite).

**Суть решения:** Создается общий базовый класс `Component`. И "Листья", и "Контейнеры" наследуются от него. Благодаря полиморфизму, если клиент вызывает метод `execute()` у Контейнера, тот просто перебирает всех своих детей и вызывает `execute()` у них.

## 🛠 Архитектура (Роли)
1. **Component (Компонент):** Общий интерфейс или абстрактный класс. Описывает операции, общие для простых и сложных элементов дерева (например, `draw()`, `getCost()`). Часто содержит метод для ссылки на родителя (parent).
2. **Leaf (Лист):** Базовый элемент дерева, у которого **нет детей**. Выполняет реальную работу.
3. **Composite / Container (Контейнер):** Элемент, имеющий потомков (листья или другие контейнеры). Реализует методы добавления/удаления детей (`add()`, `remove()`). При вызове базового метода (например, `draw()`), делегирует работу всем своим детям.
4. **Client (Клиент):** Работает со всеми элементами через базовый интерфейс `Component`.

## 💻 Пошаговая реализация и Примеры

### Пример 1: Файловая система (Из Лабы 10 / Туториала 10)

    #include <iostream>
    #include <vector>
    #include <string>

    // 1. Общий интерфейс (Component)
    class FileSystemNode {
    protected:
        std::string name;
    public:
        FileSystemNode(std::string n) : name(n) {}
        virtual ~FileSystemNode() = default;
        
        // Общий метод для всех
        virtual void print(int depth = 0) const = 0;
        
        // ВАЖНО: В классическом GoF методы add/remove добавляют прямо в Component. 
        // Но это нарушает ISP (листьям они не нужны).
        // Поэтому лучше делать безопасную реализацию (кидать exception в базовом классе).
        virtual void add(FileSystemNode* node) { throw std::logic_error("Cannot add"); }
    };

    // 2. Лист (Leaf) - не имеет детей
    class File : public FileSystemNode {
    public:
        File(std::string name) : FileSystemNode(name) {}
        
        void print(int depth = 0) const override {
            std::cout << std::string(depth, '-') << " File: " << name << std::endl;
        }
    };

    // 3. Контейнер (Composite) - имеет детей
    class Directory : public FileSystemNode {
    private:
        std::vector<FileSystemNode*> children; // Список компонентов! (Полиморфизм)
    public:
        Directory(std::string name) : FileSystemNode(name) {}
        
        void add(FileSystemNode* node) override {
            children.push_back(node);
        }
        
        // Делегируем работу детям!
        void print(int depth = 0) const override {
            std::cout << std::string(depth, '-') << " Dir: " << name << std::endl;
            for (auto child : children) {
                child->print(depth + 2); // Рекурсия
            }
        }
    };

    // Клиентский код:
    Directory* root = new Directory("root");
    Directory* docs = new Directory("documents");
    File* f1 = new File("resume.pdf");
    
    docs->add(f1);
    root->add(docs);
    root->print(); // Одной командой печатаем ВСЁ огромное дерево!

## ⚖️ Плюсы и Минусы
**Плюсы:**
* **Полиморфизм и Рекурсия:** Позволяет легко работать с огромными древовидными структурами. Клиенту не нужно писать `if (type == DIR) ... else if (type == FILE)`.
* **Open/Closed Principle:** Можно добавлять новые типы листьев или контейнеров, не меняя существующий код клиента (он работает только через интерфейс `Component`).

**Минусы:**
* **Слишком обобщенный интерфейс.** Бывает сложно спроектировать интерфейс `Component` так, чтобы он имел смысл и для Листа, и для Контейнера (нарушение Interface Segregation Principle). Приходится писать методы `add/remove`, которые для Листа бессмысленны (кидают исключения). Лектор (Зуев) называет это дилеммой между "Safety" (не делать методы `add` в интерфейсе) и "Uniformity" (сделать интерфейс единым).

## 🔍 Как отличить от других паттернов
* **Composite vs Decorator:** И Компоновщик, и Декоратор имеют структуру, где класс ссылается на объекты своего же интерфейса (рекурсивная композиция). Но Компоновщик агрегирует **коллекцию** детей (ветвится) и просто делегирует им задачи. Декоратор имеет только **одного** ребенка (оборачивает его) и *добавляет новое поведение* до/после вызова.
