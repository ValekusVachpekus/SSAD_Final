# Strategy (Стратегия)
**Категория:** Behavioral (Поведенческий)

## 📌 Суть (Intent)
Паттерн **Strategy** определяет семейство алгоритмов, инкапсулирует каждый из них в отдельный класс и делает их взаимозаменяемыми. Это позволяет алгоритму изменяться независимо от клиентов, которые его используют.

## 🧠 Фундаментальная идея (из Лекции 10)
На лекции Евгений Зуев приводил классический пример: **Duck Lake Simulator** (Симулятор озера уток).
*   **Проблема:** Изначально был базовый класс `Duck` с методами `swim()`, `display()`, `quack()`. Появились `MallardDuck` и `ReadheadDuck`. Затем решили добавить полет `fly()`. Добавили в базу. Но потом появились резиновые утки (`RubberDuck`), которые не умеют летать, и деревянные утки, которые не умеют ни летать, ни крякать.
*   *Ошибочное решение 1:* Наследование. Переопределять `fly()` в `RubberDuck` так, чтобы он ничего не делал. Это ведет к жесткой связности и дублированию кода (maintenance hell).
*   *Ошибочное решение 2:* Интерфейсы `Flyable`, `Quackable`. Решает проблему пустого поведения, но **не позволяет переиспользовать код** (каждый класс должен сам писать реализацию полета).
*   **Правильное решение (Strategy):** Отделить то, что меняется, от того, что остается постоянным. 
    1. Создаем интерфейсы `FlyBehavior` и `QuackBehavior`.
    2. Создаем классы-реализации: `FlyWithWings`, `FlyNoWay`, `Squeak`.
    3. Класс `Duck` **агрегирует** (содержит указатели/ссылки на) эти интерфейсы.
    4. **Ключевой принцип ООП:** *Prefer composition over inheritance* (Предпочитайте композицию наследованию).

## 🛠 Архитектура (Роли)
1. **Context (Контекст):** Класс, содержащий ссылку на объект `Strategy`. Контекст не знает, какая именно стратегия используется, он просто вызывает метод интерфейса. (В примере с утками это класс `Duck`).
2. **Strategy (Интерфейс):** Общий интерфейс для всех вариантов алгоритма.
3. **Concrete Strategies (Конкретные стратегии):** Классы, реализующие различные алгоритмы (`NaviHarvestStrategy`, `HumanHarvestStrategy` и т.д.).

## 💻 Пошаговая реализация и Примеры

### Пример 1: Утки (Из лекции)

    class FlyBehavior {
    public:
        virtual void fly() = 0;
    };

    class FlyWithWings : public FlyBehavior {
    public:
        void fly() override { cout << "I'm flying!" << endl; }
    };

    class Duck {
    protected:
        FlyBehavior* flyBehavior; // Указатель на стратегию
    public:
        void setFlyBehavior(FlyBehavior* fb) { 
            flyBehavior = fb; // Динамическая смена стратегии!
        }
        void performFly() { 
            flyBehavior->fly(); // Делегирование
        }
    };

### Пример 2: Сортировка (Из Туториала 10)

В туториале был показан классический пример сортировки массива. Контекст (`SortContext`) не знает, как сортировать. Он просто вызывает `strategy.sort(data)`. Клиент может передать `AscendingSortStrategy` или `DescendingSortStrategy` прямо во время выполнения.

### Пример 3: Пандора - Сбор ресурсов и Атака (Лаба 10 и Задание)
В задании Pandora существа (Navi, Human) собирают ресурсы и атакуют по-разному.
Вместо гигантского `switch(type)` внутри метода `harvest()`, мы используем Стратегию:

    // 1. Интерфейс стратегии
    class HarvestStrategy {
    public:
        virtual bool canHarvest(Resource resource) const = 0;
    };

    // 2. Конкретная стратегия для Нави
    class NaviHarvestStrategy : public HarvestStrategy {
    public:
        bool canHarvest(Resource resource) const override {
            return resource == Wood || resource == Herb;
        }
    };

    // 3. Контекст (Существо)
    class Creature {
    private:
        HarvestStrategy* harvestStrat; // Инъекция зависимости
    public:
        Creature(HarvestStrategy* hs) : harvestStrat(hs) {}
        void harvest(Resource r) {
            if (harvestStrat->canHarvest(r)) {
                // собираем
            }
        }
    };

## ⚖️ Плюсы и Минусы
**Плюсы:**
* Замена алгоритмов на лету (в рантайме) через `setStrategy()`.
* Изолирует реализацию алгоритма от классов, которые его используют (Single Responsibility Principle).
* Заменяет наследование композицией.
* Избавляет от огромных `if/else` или `switch` (Open/Closed Principle: добавляй новые стратегии, не трогая контекст).

**Минусы:**
* Клиентский код должен знать о различиях между стратегиями, чтобы выбрать подходящую.
* Усложняет код введением дополнительных интерфейсов и классов.
* Современные языки (C++11+) позволяют использовать функциональное программирование (лямбды) вместо паттерна Стратегия, избегая создания лишних классов.

## 🔍 Как отличить от других паттернов
* **Strategy vs State:** В Стратегии клиент *сам* передает Контексту нужный алгоритм, и алгоритмы обычно не меняются сами по себе. В State контекст сам (или сами состояния) инициирует *переходы* из одного состояния в другое (Draft -> Published).
* **Strategy vs Template Method:** Стратегия меняет поведение объекта *целиком* через композицию и интерфейсы в рантайме. Template Method меняет *отдельные шаги* алгоритма через наследование на этапе компиляции.
