# Bridge (Мост)
**Категория:** Structural (Структурный)

## 📌 Суть (Intent)
Паттерн **Bridge** разделяет один огромный класс (или набор тесно связанных классов) на две независимые иерархии: **Абстракцию** и **Реализацию**. Это позволяет изменять их независимо друг от друга.

## 🧠 Фундаментальная идея (из Лекции 12)
**Проблема:** Декартово произведение классов (Combinatorial Explosion).
Допустим, у нас есть абстракция `Window` (Окно) и две платформы: `Linux` и `Windows`. Мы создаем наследников: `LinuxWindow` и `WindowsWindow`. 
Затем мы решаем добавить новые типы окон: `IconWindow` (иконка) и `TextWindow` (текст). Что произойдет? 
Нам придется создать: `LinuxIconWindow`, `LinuxTextWindow`, `WindowsIconWindow`, `WindowsTextWindow`. При добавлении новой ОС (Mac) придется добавить еще 3 класса. Иерархия разрастается в геометрической прогрессии. Слишком сильная связность (Tight Coupling) на этапе компиляции.

**Решение (Мост):** Разделить систему на две части.
1. Иерархия **Абстракций** (Что это? — IconWindow, TextWindow).
2. Иерархия **Реализаций** (Как это отрисовывается в ОС? — LinuxWindowImpl, WindowsWindowImpl).
Абстракция будет хранить *указатель* на Реализацию и делегировать ей платформенно-зависимую работу.

## 🛠 Архитектура (Роли)
1. **Abstraction (Абстракция):** Определяет высокоуровневый интерфейс управления. Содержит ссылку (указатель) на объект `Implementation`.
2. **Refined Abstraction (Уточненная абстракция):** Расширяет базовую абстракцию (например, `TextWindow` добавляет методы для работы с текстом).
3. **Implementation (Реализация):** Общий интерфейс для всех платформ. Не обязан совпадать с интерфейсом `Abstraction`. Обычно `Implementation` предоставляет примитивные операции (нарисовать линию, пиксель), а `Abstraction` строит из них сложные (нарисовать окно).
4. **Concrete Implementations (Конкретные реализации):** Платформенно-зависимый код (API Windows, Linux).

## 💻 Пошаговая реализация и Примеры

### Пример 1: Музыкальный плеер (Лаба 12)
Нужно было связать Жанры музыки (Pop, Jazz, Rock) и Кодеки (MP3, WAV, FLAC). Без моста было бы `PopMP3Player`, `PopWAVPlayer` и т.д.

    // === ИЕРАРХИЯ РЕАЛИЗАЦИИ (Кодеки) ===
    class IAudioCodec {
    public:
        virtual string decode() = 0;
    };

    class MP3Codec : public IAudioCodec {
    public:
        string decode() override { return "Decoding via MP3"; }
    };

    // === ИЕРАРХИЯ АБСТРАКЦИИ (Плееры) ===
    class MusicPlayer {
    protected:
        IAudioCodec* codec; // ТОТ САМЫЙ МОСТ (Bridge)
    public:
        MusicPlayer(IAudioCodec* c) : codec(c) {}
        virtual void play() = 0;
    };

    class RockPlayer : public MusicPlayer {
    public:
        RockPlayer(IAudioCodec* c) : MusicPlayer(c) {}
        void play() override {
            cout << "Playing Rock Music... " << codec->decode() << endl;
        }
    };

    // В main():
    IAudioCodec* mp3 = new MP3Codec();
    MusicPlayer* player = new RockPlayer(mp3);
    player->play(); // Комбинируем что угодно с чем угодно в рантайме!

### Пример 2: Уведомления (Туториал 11)
В Python-коде был пример отправки `StandardNotification` и `UrgentAlert` через `EmailSender` или `SlackSender`. Абстракция (Тип алерта) форматирует текст, а Реализация (Сендер) отправляет его по сети.

## ⚖️ Плюсы и Минусы
**Плюсы:**
* Позволяет создавать платформо-независимые классы (кроссплатформенность).
* Скрывает детали реализации от клиента.
* **Open/Closed Principle:** Можно добавлять новые абстракции и новые реализации независимо друг от друга.
* **Single Responsibility Principle:** Разделяет высокоуровневую логику (Абстракция) и низкоуровневую работу платформы (Реализация).

**Минусы:**
* Усложняет код программы при применении к классу, который и так отлично справлялся со своей работой (Overengineering).

## 🔍 Как отличить от других паттернов
* **Bridge vs Adapter:** Адаптер применяется **ПОСЛЕ** того, как система спроектирована (чтобы подружить несовместимое). Мост проектируется **ДО** написания кода, чтобы позволить частям развиваться независимо.
