# Command (Команда / Действие / Транзакция)
**Категория:** Behavioral (Поведенческий)

## 📌 Суть (Intent)
Паттерн **Command** превращает запрос на выполнение действия в **отдельный объект**. Это позволяет передавать действия как аргументы методов, ставить их в очередь, логировать, а самое главное — **отменять операции (Undo/Redo)**.

## 🧠 Фундаментальная идея (из Лекции 14 и Туториала 14)
**Проблема (Пример из лекции: Универсальный пульт):** У нас есть умный пульт. Мы хотим, чтобы кнопка 1 включала Свет (`Light`), а кнопка 2 включала Кондиционер (`AirCond`). Свет включается методом `on()`, а кондиционер требует еще и установки скорости `setSpeed(20)`. Если мы пропишем вызовы прямо в класс `RemoteControl` (Пульт), то пульт будет намертво привязан к конкретным устройствам.
**Решение:** Создать общий интерфейс `Command` с методом `execute()`. Конкретные классы-команды (например, `LightOnCommand`) будут хранить внутри себя указатель на лампочку и внутри `execute()` вызывать нужные методы. Пульт будет просто дергать `slot->execute()`, не зная, что именно он включает.

## 🛠 Архитектура (Роли)
1. **Command (Интерфейс):** Декларирует метод выполнения (обычно `execute()`, часто добавляют `undo()`).
2. **Concrete Command (Конкретная команда):** Класс, реализующий действие. Он НЕ выполняет работу сам! Он хранит параметры запроса и ссылку на Получателя (Receiver), которому и делегирует работу.
3. **Receiver (Получатель):** Класс, содержащий реальную бизнес-логику (Лампочка, Текстовый редактор). Именно он делает "грязную работу".
4. **Invoker (Инициатор / Sender):** Объект, который инициирует запрос (Пульт, Кнопка в интерфейсе). Хранит ссылку на объект `Command` и вызывает его `execute()`.
5. **Client (Клиент):** Связывает всё вместе: создает Receiver, создает Command (передавая туда Receiver), создает Invoker и передает ему Command.

## 💻 Пошаговая реализация и Примеры

### Пример: Текстовый редактор с функцией UNDO (Лаба 14 / Туториал 14)

    // 1. Интерфейс Команды
    class Command {
    public:
        virtual void execute() = 0;
        virtual void undo() = 0; // Ключевая фишка паттерна!
        virtual ~Command() = default;
    };

    // 2. Получатель (Receiver) - Тот, кто реально работает с текстом
    class Editor {
    public:
        string text;
        string clipboard;
    };

    // 3. Конкретная Команда (Сохраняет бэкап состояния перед выполнением)
    class PasteCommand : public Command {
    private:
        Editor* editor;
        string backup; // Для отмены
    public:
        PasteCommand(Editor* ed) : editor(ed) {}
        
        void execute() override {
            backup = editor->text; // Сохраняем состояние ДО вставки
            editor->text += editor->clipboard; // Делаем вставку
        }
        
        void undo() override {
            editor->text = backup; // Восстанавливаем состояние!
        }
    };

    // 4. Инициатор (Invoker) - История команд
    class CommandHistory {
        vector<Command*> history;
    public:
        void executeCommand(Command* c) {
            c->execute();
            history.push_back(c);
        }
        void undoLast() {
            if (!history.empty()) {
                Command* c = history.back();
                history.pop_back();
                c->undo(); // Откат!
                delete c;
            }
        }
    };

*Тонкость (Macro Command):* В лекции упоминалось, что можно создать команду `MacroCommand`, которая внутри хранит массив (или вектор) других команд. При вызове `execute()` она пройдется циклом по всем внутренним командам. Так одной кнопкой можно запустить целую серию действий.

## ⚖️ Плюсы и Минусы
**Плюсы:**
* **Single Responsibility Principle:** Разделяет классы, инициирующие операции (Кнопки UI), от классов, выполняющих их (Бизнес-логика).
* Отложенный запуск операций, сборка в очереди (Queue) и макросы.
* Идеально подходит для реализации функций отмены/повтора (Undo/Redo).

**Минусы:**
* Усложняет код: между отправителем и получателем появляется новый слой классов. На каждое новое действие нужен свой отдельный класс.

## 🔍 Как отличить от других паттернов
* **Command vs Strategy:** Очень похожи (оба инкапсулируют поведение). Разница в **Intent (Намерении)**:
  * *Strategy:* Инкапсулирует разные **алгоритмы (КАК сделать)** для одной и той же задачи (как сортировать массив). Стратегия встроена в объект надолго.
  * *Command:* Инкапсулирует **запрос на действие (ЧТО сделать)** (нажать кнопку). Запрос — это одноразовый приказ, который можно передать по сети, отложить или отменить.
