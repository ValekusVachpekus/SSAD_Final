# Proxy (Заместитель / Посредник)
**Категория:** Structural (Структурный)

## 📌 Суть (Intent)
Паттерн **Proxy** предоставляет объект-заменитель (суррогат) для другого объекта. Прокси **контролирует доступ** к оригинальному объекту, позволяя выполнять дополнительные действия (проверка прав, кэширование, логирование) *до* или *после* того, как запрос дойдет до оригинала.

## 🧠 Фундаментальная идея (из Лекции 11 и Туториала 11)
Зачем нам "представитель"?
1. **Virtual Proxy (Ленивая инициализация):** Оригинальный объект "тяжелый" (например, загружает видео на 5 ГБ в ОЗУ). Мы не хотим создавать его при старте программы. Прокси создается быстро, а тяжелый объект загрузит только тогда, когда клиент реально вызовет метод `play()`.
2. **Protection Proxy (Защита):** Проверка прав доступа. Прокси проверяет, есть ли у пользователя права `isAdmin`, прежде чем передать команду на удаление БД оригинальному объекту.
3. **Smart Reference:** Добавление функционала к сырым указателям (подсчет ссылок в `shared_ptr` — это классический смарт-прокси!).
4. **Remote Proxy:** Оригинальный объект находится на другом сервере. Прокси берет на себя сетевые вызовы (REST API / RPC).

## 🛠 Архитектура (Роли)
1. **Service Interface (Интерфейс):** Общий интерфейс для Реального объекта и Прокси. Клиент не должен знать, с кем он говорит.
2. **Service (Реальный объект / Adaptee):** Класс, содержащий реальную, тяжелую или защищаемую бизнес-логику.
3. **Proxy (Заместитель):** Хранит ссылку на Реальный объект (часто создает его сам, управляя его жизненным циклом). Реализует тот же интерфейс. Делает проверки и делегирует вызов Реальному объекту.
4. **Client (Клиент):** Работает с прокси через общий интерфейс.

## 💻 Пошаговая реализация и Примеры

### Пример 1: Ленивая загрузка изображения (Туториал 11)

    // Общий интерфейс
    class Image {
    public:
        virtual void display() = 0;
    };

    // Тяжелый объект
    class RealImage : public Image {
        string filename;
    public:
        RealImage(string filename) : filename(filename) {
            cout << "Loading heavy image from disk: " << filename << endl; 
            // Представим, что тут программа висит 3 секунды
        }
        void display() override { cout << "Displaying " << filename << endl; }
    };

    // ПРОКСИ
    class ProxyImage : public Image {
    private:
        RealImage* realImage; // Указатель на реальный объект
        string filename;
    public:
        ProxyImage(string filename) : filename(filename), realImage(nullptr) {
            // МЫ НЕ ГРУЗИМ КАРТИНКУ ТУТ! Конструктор отрабатывает мгновенно.
        }
        
        void display() override {
            if (realImage == nullptr) {
                // Ленивая инициализация (Lazy Initialization)
                realImage = new RealImage(filename); 
            }
            realImage->display(); // Делегирование
        }
    };

    // В main:
    Image* img = new ProxyImage("test.jpg");
    // Картинка еще не загружена. Программа не зависла.
    img->display(); // Вот только сейчас начнется загрузка и вывод.

### Пример 2: Защита доступа (Лаба 11)
В Лабе 11 был `SecureDocumentProxy`.
Если юзер имеет роль "Admin", прокси перенаправляет вызов `display()` в `RealDocument`.
Если роль "Guest", прокси выводит в консоль "Access Denied" и *не вызывает* реальный объект.

## ⚖️ Плюсы и Минусы
**Плюсы:**
* Контроль над тяжелыми или опасными объектами, когда клиентам нельзя давать прямой доступ.
* Жизненный цикл (создание/удаление) реального объекта можно жестко контролировать внутри Прокси.
* Прокси работает даже тогда, когда реального объекта еще не существует.
* **Open/Closed Principle:** Прокси вводится без изменения кода реального класса.

**Минусы:**
* Увеличивает количество классов (усложняет код).
* Вызов методов может задерживаться (overhead на проверки, логирование или сетевые запросы внутри Прокси).

## 🔍 Как отличить от других паттернов
* **Proxy vs Decorator:** Оба имеют одинаковый интерфейс с оборачиваемым объектом. 
  * В **Декораторе** обернутый объект передается снаружи (через `new`), так как цель — *собрать "матрешку" из разных поведений*. 
  * В **Прокси** заместитель обычно *сам* создает и контролирует жизненный цикл своего реального объекта, так как его цель — *контролировать доступ*.
* **Proxy vs Facade:** Прокси предоставляет *тот же самый* интерфейс, что и один конкретный объект. Фасад предоставляет *новый, упрощенный* интерфейс для целой подсистемы классов.
