# Visitor (Посетитель) - САМЫЙ СЛОЖНЫЙ ПАТТЕРН!
**Категория:** Behavioral (Поведенческий)

## 📌 Суть (Intent)
Паттерн **Visitor** позволяет добавлять новые операции к иерархии классов, **не изменяя код самих этих классов**. 

## 🧠 Фундаментальная идея (из Лекции 13) - ВНИМАНИЕ!
Это прямое продолжение паттерна `Interpreter`/`Composite`.
**Проблема:** У нас есть AST-дерево из узлов `AddExp`, `OrExp`, `Constant`. В паттерне Интерпретатор мы добавили метод `evaluate()`. А что если нам нужно написать Оптимизатор кода? Валидатор кода? Генератор JSON/XML? Добавлять методы `optimize()`, `verify()`, `printJSON()` в **каждый** класс узла? Классы раздуются (нарушение SRP). Перекомпилировать придется весь проект!

**Решение (Visitor):** Мы выносим все эти операции в отдельные классы (Посетители).
1. Узлы получают всего один метод навсегда: `accept(Visitor* v)`.
2. Мы создаем класс `XMLVisitor` и помещаем в него все методы экспорта (`visit(AddExp)`, `visit(Constant)`).
3. **Double Dispatch (Двойная диспетчеризация):** Мы не можем просто вызвать `visitor->visit(node)`, потому что компилятор C++ не поддерживает динамическое связывание по аргументам функции (он не знает в рантайме, какой именно метод `visit` вызвать). Поэтому узел вызывает `v->visit(this)`. Ключевое слово `this` имеет конкретный тип узла на этапе компиляции, и компилятор точно знает, какой метод вызвать!

## 🛠 Архитектура (Роли)
1. **Visitor (Интерфейс посетителя):** Объявляет набор методов `visit()`, принимающих в качестве аргумента конкретные классы элементов иерархии (по одному методу на каждый класс).
2. **Concrete Visitor (Конкретный посетитель):** Класс, реализующий конкретный алгоритм для всей иерархии (например, `XMLVisitor` или `BonusCalculatorVisitor`).
3. **Element (Интерфейс элемента):** Объявляет метод `accept(Visitor v)`, который нужен для "впуска" посетителя.
4. **Concrete Element (Конкретный элемент):** Реализует метод `accept`. Имплементация всегда выглядит так: `v->visit(this)`.

## 💻 Пошаговая реализация и Примеры

### Пример: Расчет скидок (Лаба 13) / Экспорт Фигур в XML

    // Предварительное объявление классов элементов
    class Dot;
    class Circle;

    // 1. ИНТЕРФЕЙС VISITOR (Должен знать обо ВСЕХ элементах)
    class Visitor {
    public:
        virtual void visitDot(Dot* d) = 0;
        virtual void visitCircle(Circle* c) = 0;
    };

    // 2. ИНТЕРФЕЙС ЭЛЕМЕНТА
    class Shape {
    public:
        virtual void accept(Visitor* v) = 0; // "Дверь" для посетителя
    };

    // 3. КОНКРЕТНЫЕ ЭЛЕМЕНТЫ
    class Dot : public Shape {
    public:
        void accept(Visitor* v) override {
            // Элемент говорит: "Эй, посетитель, я - Точка, сделай со мной что-то"
            v->visitDot(this); 
        }
    };

    class Circle : public Shape {
    public:
        void accept(Visitor* v) override {
            v->visitCircle(this); // Двойная диспетчеризация!
        }
    };

    // 4. КОНКРЕТНЫЙ ПОСЕТИТЕЛЬ (Выносит бизнес-логику в себя!)
    class XMLExportVisitor : public Visitor {
    public:
        void visitDot(Dot* d) override {
            cout << "<dot>Экспорт Точки</dot>\n";
        }
        void visitCircle(Circle* c) override {
            cout << "<circle>Экспорт Круга</circle>\n";
        }
    };

    // Клиент:
    Shape* shape = new Circle();
    Visitor* xmlExporter = new XMLExportVisitor();
    
    // Хочу экспортировать фигуру в XML:
    shape->accept(xmlExporter); 
    // Под капотом: Circle::accept -> xmlExporter->visitCircle(this) -> cout << "<circle>..."

## ⚖️ Плюсы и Минусы
**Плюсы:**
* **Open/Closed Principle (Огромный плюс):** Мы можем добавить миллион новых поведений (JSON-экспорт, Валидатор, Калькулятор), просто создав новый класс `ConcreteVisitor`. Нам **не придется трогать классы элементов (Shape, Dot, Circle)** вообще!
* **Single Responsibility Principle:** Бизнес-логика, связанная с XML, находится только в классе `XMLExportVisitor`, а не размазана по `Dot` и `Circle`.

**Минусы:**
* **Нарушение OCP в обратную сторону:** Паттерн великолепен, пока мы добавляем новые *операции*. Но если нам понадобится добавить новый *элемент* (например, `Rectangle`), нам придется пойти в класс `Visitor` и добавить метод `visitRectangle(Rectangle* r)`. Это сломает интерфейс и потребует обновления ВСЕХ существующих классов посетителей.

## 🔍 Как отличить от других паттернов
* **Visitor vs Iterator:** Итератор просто ходит по коллекции (не зная её структуры). Визитор выполняет мощные, специфичные для каждого узла операции на основе его типа, используя механизм двойной диспетчеризации (`accept(this)` -> `visit(ConcreteType)`).
