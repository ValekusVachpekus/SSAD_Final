# Observer (Наблюдатель / Издатель-Подписчик)
**Категория:** Behavioral (Поведенческий)

## 📌 Суть (Intent)
Паттерн **Observer** определяет механизм подписки: он позволяет объектам-наблюдателям (Subscribers/Observers) подписываться на события, происходящие в объекте-издателе (Publisher/Subject). При изменении состояния издатель автоматически оповещает всех своих подписчиков.

## 🧠 Фундаментальная идея (из Лекции 14)
Отношение "Один-ко-Многим" (One-to-Many).
**Проблема:** В соцсети есть Владелец группы (Publisher) и Подписчики (Subscribers). Когда выходит новый пост, подписчики не должны каждую секунду опрашивать сервер: *"Вышел пост? Вышел пост?"* (Это тратит CPU).
**Решение:** Разделить систему. Publisher должен иметь список своих подписчиков. Когда выходит пост, Publisher бежит циклом по списку и вызывает у каждого подписчика метод `update(news)`.

## 🛠 Архитектура (Роли)
1. **Publisher / Subject (Издатель):** Класс, состояние которого представляет интерес для других объектов. Хранит массив (список) ссылок на объекты подписчиков. Содержит методы для добавления `attach(Observer*)`, удаления `detach()` и оповещения `notify()`.
2. **Subscriber / Observer (Интерфейс Наблюдателя):** Интерфейс, который заставляет всех подписчиков иметь метод `update()`.
3. **Concrete Subscribers (Конкретные подписчики):** Различные классы, которые реагируют на оповещения по-своему (Один пишет лог в консоль, второй шлет Email).
4. **Client:** Создает Издателя и Подписчиков, а затем регистрирует подписчиков в Издателе (`publisher.attach(sub1)`).

## 💻 Пошаговая реализация и Примеры

### Пример: Текстовый редактор с Логгером (Лаба 14 / Assignment 4)
Нам нужно эффективно подсчитывать статистику (например, `COUNT` и `LOAD` в Assignment 4), не проходясь циклом по всем элементам каждый раз.

    // 1. ИНТЕРФЕЙС НАБЛЮДАТЕЛЯ
    class IEventListener {
    public:
        // Метод, который дернет Издатель
        virtual void update(const string& eventType, const string& file) = 0;
    };

    // 2. ИЗДАТЕЛЬ (Хранит мапу событий и подписчиков)
    class EventManager {
        unordered_map<string, vector<IEventListener*>> listeners;
    public:
        void subscribe(string event, IEventListener* listener) {
            listeners[event].push_back(listener);
        }
        
        void notify(string event, string file) {
            for (auto listener : listeners[event]) {
                listener->update(event, file); // Рассылаем оповещение!
            }
        }
    };

    // КОНКРЕТНЫЙ ИЗДАТЕЛЬ (Использует EventManager под капотом)
    class Editor {
    public:
        EventManager events;
        
        void saveFile(string filename) {
            // ... Логика сохранения ...
            events.notify("save", filename); // Громко кричим: Я сохранился!
        }
    };

    // 3. КОНКРЕТНЫЕ ПОДПИСЧИКИ
    class EmailListener : public IEventListener {
    public:
        void update(const string& eventType, const string& file) override {
            cout << "Sending Email: File " << file << " was " << eventType << endl;
        }
    };

    // Клиент:
    Editor editor;
    EmailListener emailAlert;
    
    // Подписываемся!
    editor.events.subscribe("save", &emailAlert);
    
    // Теперь редактор ничего не знает об EmailAlert, 
    // но при сохранении EmailAlert сработает!
    editor.saveFile("report.pdf"); 

## ⚖️ Плюсы и Минусы
**Плюсы:**
* **Open/Closed Principle:** Можно вводить в систему новые типы подписчиков без изменения кода Издателя. Издатель связан с ними только через абстрактный интерфейс.
* Отношения устанавливаются динамически (в рантайме) — можно подписываться и отписываться на лету (`unsubscribe`).

**Минусы:**
* Подписчики оповещаются в **случайном порядке** (порядок в списке не гарантируется и не должен влиять на логику).
* При плохом дизайне возможны **утечки памяти**, если Подписчик был удален, но не отписался от Издателя (у Издателя останется висячий указатель).

## 🔍 Как отличить от других паттернов
* **Observer vs Mediator:** Разница в архитектуре взаимодействия.
  * В **Observer** поток данных идет в одну сторону: от Publisher к Subscribers (Один-ко-Многим).
  * В **Mediator** поток данных идет от *Всех* ко *Всем*, но через центральный узел (Много-к-Медиатору-ко-Многим).
