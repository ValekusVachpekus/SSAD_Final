# Facade (Фасад)
**Категория:** Structural (Структурный)

## 📌 Суть (Intent)
Паттерн **Facade** предоставляет **простой (упрощенный) интерфейс** к сложной системе классов, библиотеке или фреймворку.

## 🧠 Фундаментальная идея (из Лекции 11 и Туториала 11)
**Проблема (Лекция 11 - Компилятор):** Любой компилятор — это чудовищно сложная система. У него есть подсистемы: `Scanner` (разбивает текст на токены), `Parser` (строит AST-дерево), `CodeGenerator` (генерирует машинный код) и `Optimizer`. Если программист просто хочет скомпилировать файл, ему придется писать десятки строк кода, инициализируя каждый модуль по очереди и передавая данные между ними. Это ведет к жесткому связыванию (Tight Coupling) клиента с потрохами библиотеки.

**Решение:** Создать один класс `Compiler` (это и есть Фасад). Внутри себя он скроет всю сложную логику взаимодействия подсистем. Наружу он выставит всего один метод: `compile()`.

## 🛠 Архитектура (Роли)
1. **Facade (Фасад):** Класс, предоставляющий удобный доступ к определенной части функциональности подсистемы. Он знает, куда направить запрос клиента и какие модули задействовать.
2. **Additional Facade (Дополнительный фасад - опционально):** Если основной фасад разрастается (превращаясь в God Object), логику можно разделить на несколько фасадов.
3. **Complex Subsystem (Сложная подсистема):** Десятки классов. Они **ничего не знают о существовании Фасада** — для них это просто еще один клиент, который их вызывает.
4. **Client (Клиент):** Работает с Фасадом вместо прямого вызова объектов подсистемы.

## 💻 Пошаговая реализация и Примеры

### Пример 1: Умный дом (Лаба 11)
В доме есть умный свет (`Light`), термостат (`Thermostat`) и камеры (`SecurityCamera`).
Мы хотим сделать сценарий "Ухожу из дома", который выключит свет, переведет температуру в эко-режим и включит камеры.

    class Light { public: void off() { cout << "Lights off\n"; } };
    class Thermostat { public: void setEco() { cout << "Thermostat to eco\n"; } };
    class SecurityCamera { public: void activate() { cout << "Cameras ON\n"; } };

    // ФАСАД
    class SmartHomeFacade {
    private:
        Light light;
        Thermostat thermostat;
        SecurityCamera camera;
    public:
        // Упрощенный интерфейс для клиента
        void leavingHome() {
            cout << "Running Leaving Home scenario...\n";
            light.off();
            thermostat.setEco();
            camera.activate();
        }
    };

    // В main():
    SmartHomeFacade home;
    home.leavingHome(); // Клиенту не нужно дергать 3 разных устройства!

### Пример 2: Конвертер Видео (Туториал 11)
Использование сложной библиотеки конвертации (AudioMixer, BitrateReader, CodecFactory). Фасад `VideoConversionFacade` принимает строку `filename` и прячет внутри себя все `new MPEG4CompressionCodec()` и вызовы потоков битов.

## ⚖️ Плюсы и Минусы
**Плюсы:**
* Изолирует клиентский код от сложной подсистемы. 
* Снижает зависимости (Decoupling): если библиотека конвертации видео обновится (например, `BitrateReader` изменит сигнатуру), вы перепишете только класс `Facade`. Весь остальной код программы (тысячи файлов) не пострадает.

**Минусы:**
* Фасад может превратиться в **God Object** (Объект-всезнайку), который привязан ко всем классам программы и раздут на тысячи строк.

## 🔍 Как отличить от других паттернов
* **Facade vs Adapter:** Адаптер заставляет старый объект работать с *новым интерфейсом*, который ожидает клиент (и обычно оборачивает только *один* объект). Фасад создает *совершенно новый интерфейс*, чтобы упростить работу с *группой* объектов.
* **Facade vs Mediator:** Оба организуют взаимодействие между множеством объектов. Но:
  * В **Facade** подсистемы *не знают* о фасаде, они работают друг с другом напрямую. Фасад просто задает им вектор работы сверху вниз.
  * В **Mediator** компоненты *знают* о медиаторе и общаются *только* через него (жесткая двусторонняя связь).
