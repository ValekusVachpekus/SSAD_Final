# Decorator (Декоратор)
**Категория:** Structural (Структурный)

## 📌 Суть (Intent)
Паттерн **Decorator** позволяет **динамически** (во время выполнения программы) добавлять объектам новые поведения и обязанности, помещая их внутрь специальных объектов-оберток (wrappers).

## 🧠 Фундаментальная идея (из Лекции 11)
**Проблема (Пример: Текстовый редактор):** Есть окно `TextView`. Мы хотим добавить к нему полосу прокрутки (`ScrollDecorator`) или рамку (`BorderDecorator`). 
*   *Ошибочное решение:* Использовать наследование. Сделать класс `ScrolledWindow`, затем `BorderedWindow`. А что если нужно окно и с рамкой, и со скроллом? `BorderedScrolledWindow`? Возникает комбинаторный взрыв классов. Наследование — это статика (compile-time). Мы не можем убрать рамку у окна прямо во время работы программы.
*   **Решение (Decorator):** Создать класс-обертку, который имеет тот же интерфейс, что и окно. Обертка будет хранить указатель на оригинальное окно, выполнять свою доп. задачу (рисовать рамку) и вызывать метод `draw()` у оригинального окна. Мы можем вкладывать обертки друг в друга как матрешки!

## 🛠 Архитектура (Роли)
1. **Component (Интерфейс компонента):** Общий интерфейс для базовых объектов и всех декораторов.
2. **Concrete Component (Конкретный компонент):** Базовый класс, который мы будем оборачивать (например, "чистый" текст).
3. **Base Decorator (Базовый декоратор):** Имеет поле типа `Component`, ссылающееся на оборачиваемый объект. Делегирует ему все вызовы.
4. **Concrete Decorators (Конкретные декораторы):** Наследуются от базового декоратора. Выполняют свою логику (добавление тегов `<b>`, шифрование, сжатие) ДО или ПОСЛЕ вызова метода обернутого объекта.

## 💻 Пошаговая реализация и Примеры

### Пример 1: Окна (Из Лекции 11)
    
    // 1. Общий интерфейс
    class Component {
    public:
        virtual void draw() = 0;
    };

    // 2. Базовый объект
    class TextView : public Component {
    public:
        void draw() override { cout << "Drawing Text\n"; }
    };

    // 3. Базовый декоратор
    class Decorator : public Component {
    protected:
        Component* wrappee; // Ссылка на обернутый объект
    public:
        Decorator(Component* c) : wrappee(c) {}
        void draw() override { wrappee->draw(); } // Делегирование
    };

    // 4. Конкретный декоратор
    class BorderDecorator : public Decorator {
    public:
        BorderDecorator(Component* c) : Decorator(c) {}
        void draw() override {
            cout << "Drawing Border\n"; // Моя доп. функциональность
            Decorator::draw();          // Вызов оригинального поведения
        }
    };

    // В main:
    Component* window = new TextView();
    Component* borderedWindow = new BorderDecorator(window); // Матрешка 1
    borderedWindow->draw(); // Выведет: Drawing Border -> Drawing Text

### Пример 2: Форматирование текста (Лаба 11)
Нужно было обернуть `PlainText` в теги (Bold, Italic).
Код выглядит так: `Text* t = new BoldDecorator(new ItalicDecorator(new PlainText("Hello")));`
При вызове `t->write()` матрешка раскручивается изнутри наружу: `<b><i>Hello</i></b>`.

## ⚖️ Плюсы и Минусы
**Плюсы:**
* БОЛЬШАЯ гибкость по сравнению с наследованием. Поведение добавляется и удаляется в рантайме.
* Можно комбинировать несколько поведений (навешивать хоть 10 слоев оберток).
* **Single Responsibility:** Можно разбить один гигантский класс с сотнями настроек на много маленьких классов-декораторов.

**Минусы:**
* Тяжело удалить конкретную обертку из середины стека (середину матрешки).
* Код создания матрешки выглядит уродливо: `new A(new B(new C(new Object())))`. (Обычно решается паттерном Factory/Builder).
* Поведение декоратора не должно зависеть от порядка оберток (но на практике этого сложно добиться).

## 🔍 Как отличить от других паттернов
* **Decorator vs Proxy:** ОЧЕНЬ ПОХОЖИ структурно (оба хранят ссылку на оригинальный объект и реализуют тот же интерфейс). **Разница в Intent:** 
  * Proxy *контролирует доступ* к объекту (жизненным циклом он часто управляет сам, создавая оригинал внутри себя). 
  * Decorator *добавляет новые обязанности* (всегда принимает готовый объект снаружи через конструктор).
* **Decorator vs Adapter:** Адаптер меняет интерфейс объекта, Декоратор сохраняет интерфейс прежним.
