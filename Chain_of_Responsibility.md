# Chain of Responsibility (Цепочка обязанностей)
**Категория:** Behavioral (Поведенческий)

## 📌 Суть (Intent)
Паттерн **Chain of Responsibility** позволяет передавать запросы последовательно по цепочке обработчиков. Каждый обработчик решает, может ли он сам обработать запрос, или его нужно передать дальше (следующему обработчику по цепочке).

## 🧠 Фундаментальная идея (из Лекции 14)
**Проблема (Пример из лекции: Система контекстной справки в UI):** Пользователь нажимает `F1`. Справка должна зависеть от того, где находится курсор. Если курсор на кнопке — показываем справку для кнопки. Если у кнопки нет справки — просим показать справку у Панели. Если у Панели нет — у Окна. Если у Окна нет — показываем общую справку приложения. 
Если использовать жесткое кодирование, Окну придется знать обо всех кнопках и панелях.
**Решение:** Выстроить объекты (Кнопка -> Панель -> Окно) в цепочку. Запрос кидается в самое начало (в Кнопку). Если кнопка не может его обработать (нет текста справки), она пересылает его "родителю" по цепочке.

## 🛠 Архитектура (Роли)
1. **Handler (Интерфейс Обработчика):** Объявляет метод для обработки запроса (например, `handle()`) и метод для установки следующего обработчика (`setNext()`).
2. **Base Handler (Базовый обработчик):** (Опционально) Реализует логику по умолчанию: "если есть следующий обработчик, передай запрос ему". Это избавляет от дублирования кода (`if (next) next->handle(req);`).
3. **Concrete Handlers (Конкретные обработчики):** Реализуют код обработки. Принимают решение: обработать запрос (и возможно завершить цепочку) ИЛИ передать его дальше.
4. **Client (Клиент):** Создает объекты-обработчики и связывает их в цепь в рантайме.

## 💻 Пошаговая реализация и Примеры

### Пример: Авторизация и Логирование (Лаба 14 / Assignment 4)

    // 1. Интерфейс и Базовый класс (Избавляет от дублирования)
    class Handler {
    protected:
        Handler* next = nullptr;
    public:
        void setNext(Handler* n) { next = n; }
        
        // Поведение по умолчанию: проброс дальше
        virtual bool handle(const string& request) {
            if (next != nullptr) {
                return next->handle(request);
            }
            return true; // Цепочка пройдена успешно
        }
    };

    // 2. Звено 1: Проверка аутентификации
    class AuthHandler : public Handler {
    public:
        bool handle(const string& request) override {
            if (request == "hacker") {
                cout << "Auth failed!" << endl;
                return false; // ПРЕРЫВАЕМ ЦЕПОЧКУ! Дальше запрос не пойдет.
            }
            cout << "Auth OK. Passing to next..." << endl;
            return Handler::handle(request); // Делегируем базовому классу проброс
        }
    };

    // 3. Звено 2: Проверка роли (Парсер)
    class RoleHandler : public Handler {
    public:
        bool handle(const string& request) override {
            if (request == "admin") {
                cout << "Role: Admin! Request completed." << endl;
                return true; // МЫ ОБРАБОТАЛИ. Цепочка завершается успешно.
            }
            return Handler::handle(request); // Передаем дальше
        }
    };

    // Клиент
    Handler* auth = new AuthHandler();
    Handler* role = new RoleHandler();
    auth->setNext(role); // Строим конвейер: auth -> role

    auth->handle("admin"); // Вызов пройдет через Auth и остановится в Role.

## ⚖️ Плюсы и Минусы
**Плюсы:**
* **Decoupling:** Отправитель запроса не знает, кто именно в цепочке обработает его запрос.
* **Single Responsibility / Open-Closed:** Обработчики независимы друг от друга. Можно менять порядок цепочки и добавлять новые фильтры прямо во время работы программы (`runtime`).

**Минусы:**
* Запрос может остаться **необработанным**, если он дойдет до конца цепочки, а никто его не перехватит.
* Если цепочка слишком длинная, отладка (дебаг) может стать кошмаром (трудно понять, в каком звене запрос потерялся).

## 🔍 Как отличить от других паттернов
* Часто используется вместе с **Composite (Компоновщик)**. В интерфейсе окон (Composite) события всплывают (bubbling) от дочерних элементов (кнопок) к родительским (окнам) — это и есть Chain of Responsibility.
* В паттерне Command один запрос выполняется *одним* получателем. В Chain of Responsibility запрос идет *по конвейеру* до тех пор, пока кто-то его не обработает.
