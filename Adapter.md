# Adapter (Адаптер / Wrapper)
**Категория:** Structural (Структурный)

## 📌 Суть (Intent)
Паттерн **Adapter** позволяет объектам с несовместимыми интерфейсами работать вместе. Он действует как переходник (wrapper), переводя вызовы из одного интерфейса в другой.

## 🧠 Фундаментальная идея (из Лекции 10)
На лекции рассматривался пример с розетками (вилка стандарта ЕС и розетка стандарта США не подходят друг другу, нужен адаптер).
**Программный пример из лекции:** Снова утки! У нас есть симулятор `Duck`, который ожидает вызовы `quack()` и `fly()`. Внезапно мы хотим добавить индюшку `Turkey`, которая имеет методы `gobble()` и летает только на короткие дистанции (`fly()`).
Мы не можем изменить класс `Turkey` (это может быть сторонняя закрытая библиотека или legacy-код). Мы создаем класс `TurkeyAdapter`, который реализует интерфейс `Duck`, но внутри содержит объект `Turkey` и транслирует вызов `quack()` в `turkey.gobble()`.

## 🛠 Архитектура (Роли)
1. **Client (Клиент):** Ожидает объекты с определенным интерфейсом (например, ожидает `Duck`).
2. **Target Interface (Целевой интерфейс):** Тот интерфейс, с которым умеет работать Клиент (`Duck`).
3. **Adaptee (Адаптируемый класс / Сервис):** Сторонний, legacy или сложный класс, чей интерфейс несовместим с Клиентом (`Turkey`).
4. **Adapter (Адаптер):** Класс, который реализует `Target Interface` и оборачивает `Adaptee`.

## 💻 Пошаговая реализация и Примеры

### Пример 1: Индюшки под видом Уток (Из Лекции 10)

    class Duck { // Target Interface
    public:
        virtual void quack() = 0;
        virtual void fly() = 0;
    };

    class Turkey { // Adaptee (Несовместимый интерфейс)
    public:
        void gobble() { cout << "Gobble gobble"; }
        void fly() { cout << "I'm flying a short distance"; }
    };

    // Адаптер: наследуется от Target, внутри содержит Adaptee
    class TurkeyAdapter : public Duck {
    private:
        Turkey* turkey;
    public:
        TurkeyAdapter(Turkey* t) : turkey(t) {}
        
        void quack() override {
            turkey->gobble(); // Перевод вызова!
        }
        void fly() override {
            for(int i=0; i<5; i++) {
                turkey->fly(); // Адаптация логики полета
            }
        }
    };

*Тонкость из лекции:* Адаптер может быть реализован через **Множественное наследование** (в C++). Вместо хранения указателя, `class TurkeyAdapter : public Duck, private Turkey`. Слово `private` важно, чтобы методы `Turkey` не торчали наружу.

### Пример 2: Payment Gateway (Из Лабы 10)
Система магазина должна работать с разными платежными шлюзами (PayPal, Stripe). У них абсолютно разные API.
1. Создаем общий `interface IPaymentGateway { pay(); }`.
2. Создаем `class PayPalAdapter : public IPaymentGateway`.
3. Внутри метода `pay()` адаптера вызываем `paypalApi.processTransaction()`.

## ⚖️ Плюсы и Минусы
**Плюсы:**
* **Single Responsibility Principle:** Разделяет код конвертации интерфейсов от основной бизнес-логики программы.
* **Open/Closed Principle:** Можно вводить в программу новые типы адаптеров для сторонних библиотек, не ломая существующий клиентский код.

**Минусы:**
* Увеличивает общую сложность кода за счет введения новых классов и интерфейсов. Иногда проще изменить сам класс `Adaptee`, если у вас есть к нему доступ.

## 🔍 Как отличить от других паттернов
* **Adapter vs Bridge:** Разница в "Intent" (намерении). Адаптер создается **ПОСЛЕ** того, как классы уже написаны (чтобы "подружить" legacy-код с новым). Мост (Bridge) проектируется **ДО** написания кода, чтобы позволить абстракции и реализации развиваться независимо.
* **Adapter vs Proxy:** Proxy предоставляет *тот же самый* интерфейс к объекту. Адаптер предоставляет *другой* интерфейс к объекту.
* **Adapter vs Decorator:** Декоратор добавляет новое поведение без изменения интерфейса. Адаптер меняет интерфейс, ничего не добавляя к поведению по сути.
