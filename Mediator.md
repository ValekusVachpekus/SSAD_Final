# Mediator (Посредник / Intermediary)
**Категория:** Behavioral (Поведенческий)

## 📌 Суть (Intent)
Паттерн **Mediator** уменьшает хаотичные (спагетти) зависимости между объектами. Он заставляет объекты общаться друг с другом не напрямую, а **только через объект-посредник**.

## 🧠 Фундаментальная идея (из Лекции 15)
**Проблема (Пример: Текстовый чат):** Представь систему чата. Есть `User A`, `User B` и `User C`. Если `A` хочет отправить сообщение, он должен иметь прямые ссылки (указатели) на `B` и `C`. В итоге каждый объект класса `User` знает обо всех остальных `User`. Паутина связей сводит с ума, код невозможно поддерживать. Компонент `User` нельзя выдрать из кода и использовать в другом проекте, так как он жестко связан (Tight Coupling) с другими юзерами.
**Решение:** Ввести класс `ChatRoom` (Посредник). Пользователь знает **ТОЛЬКО** о медиаторе. Пользователь говорит медиатору: «Отправь это всем». Медиатор берет сообщение и сам вызывает методы у остальных пользователей.

## 🛠 Архитектура (Роли)
1. **Mediator (Интерфейс):** Декларирует методы для связи с компонентами (например, `notify(sender, event)`).
2. **Concrete Mediator (Конкретный посредник):** Знает обо всех компонентах (хранит их в массиве или полях). Берет на себя логику координации: «Если пришло событие от кнопки А, нужно очистить текстовое поле Б и чекбокс С».
3. **Components (Компоненты / Коллеги):** Бизнес-объекты (`User`, `Button`, `Checkbox`). Они хранят ссылку ТОЛЬКО на интерфейс `Mediator`. Они не знают о других компонентах! Если что-то случилось (клик, отправка текста) — они просто дергают метод медиатора `mediator->notify(this)`.

## 💻 Пошаговая реализация и Примеры

### Пример: Текстовый Чат (Лекция 15)

    class User; // Forward declaration

    // 1. Интерфейс Медиатора
    class ChatMediator {
    public:
        virtual void sendMessage(string msg, User* sender) = 0;
        virtual void addUser(User* user) = 0;
    };

    // 2. Базовый Компонент
    class User {
    protected:
        ChatMediator* mediator; // Связь ТОЛЬКО с медиатором
        string name;
    public:
        User(ChatMediator* m, string n) : mediator(m), name(n) {}
        
        virtual void send(string msg) {
            cout << name << " sends: " << msg << endl;
            mediator->sendMessage(msg, this); // Просим посредника!
        }
        virtual void receive(string msg) {
            cout << name << " receives: " << msg << endl;
        }
    };

    // 3. Конкретный Медиатор
    class ChatRoom : public ChatMediator {
    private:
        vector<User*> users; // Хранит всех юзеров
    public:
        void addUser(User* u) override { users.push_back(u); }
        
        void sendMessage(string msg, User* sender) override {
            for (User* u : users) {
                if (u != sender) { // Отправляем всем КРОМЕ отправителя
                    u->receive(msg);
                }
            }
        }
    };

    // В main():
    ChatMediator* chat = new ChatRoom();
    User* alice = new User(chat, "Alice");
    User* bob = new User(chat, "Bob");
    
    chat->addUser(alice); chat->addUser(bob);
    alice->send("Hi Bob!"); // Alice не знает про Bob'а напрямую!

## ⚖️ Плюсы и Минусы
**Плюсы:**
* **Single Responsibility:** Вся логика общения вынесена в одно место (Медиатор).
* **Open/Closed:** Можно добавить нового посредника, не меняя код компонентов.
* Уменьшает зацепление (Coupling): компоненты больше не зависят друг от друга, их можно переиспользовать.

**Минусы:**
* Со временем Медиатор рискует превратиться в гигантский **God Object**, который знает всё обо всех и управляет всей программой.

## 🔍 Как отличить от других паттернов
* **Mediator vs Observer (ВАЖНО!):** Это главная путаница.
  * **Observer (Издатель-Подписчик):** Связь однонаправленная (Один-ко-Многим). Подписчики сидят и слушают. Издатель просто вещает (broadcasts) всем подряд "Я изменился". Он не управляет подписчиками.
  * **Mediator:** Связь двунаправленная (Много-ко-Многим через Центр). Медиатор содержит **бизнес-логику** того, как объекты должны реагировать друг на друга. ("Если нажата кнопка X, скажи текстовому полю Y, чтобы оно очистилось"). Часто Медиатор *использует внутри себя* Observer, чтобы слушать события компонентов.
