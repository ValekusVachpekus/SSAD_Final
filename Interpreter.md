# Interpreter (Интерпретатор)
**Категория:** Behavioral (Поведенческий)

## 📌 Суть (Intent)
Паттерн **Interpreter** применяется, когда у вас есть язык (пусть даже очень простой, как математические формулы или SQL-запросы), и вам нужно определить для него грамматику, а затем написать интерпретатор, который будет читать выражения на этом языке и вычислять их.

## 🧠 Фундаментальная идея (из Лекции 13)
Паттерн строится поверх паттерна **Composite (Компоновщик)**.
Допустим, у нас есть логическое выражение: `(A or B) and (not C or D)`.
Мы можем разбить его на Абстрактное Синтаксическое Дерево (AST):
*   `AND` (Узел / Composite)
    *   `OR` (Узел) -> `A` (Лист), `B` (Лист)
    *   `OR` (Узел) -> `NOT` (Узел) -> `C` (Лист), `D` (Лист)

В паттерне `Composite` мы создали бы общую структуру (классы узлов и листьев). Паттерн `Interpreter` делает следующий шаг: он добавляет в базовый класс интерфейса (Component) метод `evaluate()`. Каждый конкретный узел переопределяет этот метод так, чтобы рекурсивно вычислять значение своих потомков.

## 🛠 Архитектура (Роли)
1. **Abstract Expression (Абстрактное выражение):** Интерфейс или абстрактный класс (как `Component` в Composite). Объявляет абстрактный метод `evaluate(Context)` (или `interpret()`).
2. **Terminal Expression (Терминальное выражение / Leaf):** Листья дерева AST. (Например, конкретные переменные `A`, `B` или числа `5`, `10`).
3. **Nonterminal Expression (Нетерминальное выражение / Composite):** Узлы дерева AST (операторы `+`, `-`, `AND`, `OR`). Они хранят ссылки на свои дочерние выражения (левое и правое) и в методе `evaluate()` рекурсивно вызывают `evaluate()` у детей, после чего применяют свою операцию.
4. **Context (Контекст):** Глобальное состояние интерпретатора. Сюда кладут, например, значения переменных (Если `A=true`, `B=false`).

## 💻 Пошаговая реализация и Примеры

### Пример: Вычисление булевых выражений (Из Лекции 13)

    // 1. Абстрактное выражение
    class BooleanExp {
    public:
        virtual bool evaluate() = 0; // Метод Интерпретатора!
        virtual ~BooleanExp() = default;
    };

    // 2. Терминальное выражение (Константа: true или false)
    class Constant : public BooleanExp {
        bool value;
    public:
        Constant(bool v) : value(v) {}
        bool evaluate() override { return value; } // Просто возвращает себя
    };

    // 3. Нетерминальные выражения (Операторы)
    class AndExp : public BooleanExp {
        BooleanExp* left;
        BooleanExp* right;
    public:
        AndExp(BooleanExp* l, BooleanExp* r) : left(l), right(r) {}
        
        bool evaluate() override {
            // Рекурсивно вычисляет детей и применяет логическое И
            return left->evaluate() && right->evaluate();
        }
    };

    class OrExp : public BooleanExp {
        BooleanExp* left;
        BooleanExp* right;
    public:
        OrExp(BooleanExp* l, BooleanExp* r) : left(l), right(r) {}
        bool evaluate() override {
            return left->evaluate() || right->evaluate();
        }
    };

    // В main() (Предполагается, что парсер уже построил AST-дерево):
    // Строим: (true OR false) AND true
    BooleanExp* ast = new AndExp(
        new OrExp(new Constant(true), new Constant(false)),
        new Constant(true)
    );
    
    // Запуск интерпретатора:
    bool result = ast->evaluate(); // Вернет true

## ⚖️ Плюсы и Минусы
**Плюсы:**
* Грамматику языка легко изменить или расширить. Достаточно добавить новый класс (например, `XorExp`), унаследованный от `BooleanExp`.
* Легко реализовать интерпретацию грамматики поверх паттерна `Composite`.

**Минусы:**
* Поддерживать грамматики со сложными правилами практически невозможно (классов станет сотни).
* Если помимо вычисления (evaluate) нужно сделать что-то еще (оптимизировать дерево, напечатать, сгенерировать машинный код), придется в каждый класс добавлять новые методы `optimize()`, `print()`. Это раздует классы. (Эту проблему решает паттерн **Visitor**!).
