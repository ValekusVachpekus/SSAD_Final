# Template Method (Шаблонный метод)
**Категория:** Behavioral (Поведенческий)

## 📌 Суть (Intent)
Паттерн **Template Method** определяет костяк (скелет) алгоритма в базовом классе, но позволяет подклассам переопределять некоторые шаги этого алгоритма, **не меняя саму его структуру**.

## 🧠 Фундаментальная идея (из Лекции 15)
**Проблема (Пример: Приготовление рецептов):** У нас есть программа, которая готовит Пиццу и Мороженое (Gelato). Процесс приготовления всегда состоит из четырех шагов:
1. Подготовить ингредиенты.
2. Приготовить (сварить/запечь).
3. Добавить украшение (гарнир) — *опциональный шаг*.
4. Подать блюдо.
Если мы напишем эти шаги в каждом классе (`PizzaRecipe`, `GelatoRecipe`), у нас будет дублирование кода (например, подача блюда часто одинаковая). А если мы захотим поменять порядок (сначала подать, потом украсить), придется менять код во всех рецептах!

**Решение (Template Method):** Создать базовый абстрактный класс `Recipe`. В нем написать метод `cookRecipeTemplate()`, который жестко задает порядок вызовов: `prepare(); cook(); if(special) garnish(); serve();`. 
Сам этот метод мы делаем **`final`** (чтобы наследники не могли сломать порядок шагов). А вот сами методы `prepare()`, `cook()` мы делаем `abstract` (виртуальными), заставляя наследников реализовать их по-своему.

## 🛠 Архитектура (Роли)
1. **Abstract Class (Абстрактный класс):** 
   * Содержит `Template Method` (Сам Шаблонный метод, обычно `final`), который задает каркас алгоритма.
   * Содержит объявление шагов алгоритма (методы `step1()`, `step2()`). Эти шаги могут быть как чисто абстрактными (`= 0`), так и иметь реализацию по умолчанию.
2. **Concrete Classes (Конкретные классы):** Наследуются от абстрактного класса и **переопределяют** (override) шаги алгоритма, но НЕ переопределяют сам Шаблонный метод.
3. **Hook (Хук / Зацепка):** Опциональный метод с пустой или дефолтной реализацией в абстрактном классе (`isSpecialDish() { return false; }`). Подклассы могут переопределить его, чтобы вклиниться в алгоритм (например, отменить шаг украшения блюда).

## 💻 Пошаговая реализация и Примеры

### Пример: Рецепты (Лекция 15)

    // 1. Абстрактный класс
    class Recipe {
    public:
        // Тот самый Шаблонный метод (Skeleton). Его нельзя переопределить!
        void cookRecipeTemplate() final {
            prepareIngredients(); // Абстрактный шаг
            cook();               // Абстрактный шаг
            
            // Хук (Hook) - подкласс может повлиять на логику
            if (isSpecialDish()) {
                garnish(); 
            }
            serve(); // Шаг с дефолтной реализацией
        }
        
    protected: // Шаги скрыты от пользователя, их вызывает только шаблонный метод!
        virtual void prepareIngredients() = 0;
        virtual void cook() = 0;
        virtual void garnish() = 0;
        
        // Дефолтная реализация
        virtual void serve() { cout << "Serve the dish\n"; }
        
        // Хук (по умолчанию false)
        virtual bool isSpecialDish() { return false; }
    };

    // 2. Конкретные классы
    class PizzaRecipe : public Recipe {
    protected:
        void prepareIngredients() override { cout << "Dough, sauce, cheese\n"; }
        void cook() override { cout << "Bake in oven\n"; }
        void garnish() override { cout << "Add basil\n"; }
        
        // Переопределили хук! Теперь Пицца будет с гарниром.
        bool isSpecialDish() override { return true; } 
    };

    // Клиент
    Recipe* pizza = new PizzaRecipe();
    pizza->cookRecipeTemplate(); // Клиент вызывает ТОЛЬКО шаблонный метод!

## ⚖️ Плюсы и Минусы
**Плюсы:**
* Избавляет от дублирования кода (выносим общий код в суперкласс).
* Защищает общую структуру алгоритма от поломок в подклассах.

**Минусы:**
* Подклассы жестко зажаты в рамки алгоритма, заданного базовым классом.
* Большое количество шагов делает шаблонный метод трудным для поддержки.
* Нарушение Liskov Substitution: если подкласс переопределит дефолтный шаг и оставит его пустым (подавит базовую логику), это может сломать общую идею.

## 🔍 Как отличить от других паттернов
* **Template Method vs Strategy (Важнейший вопрос!):** 
  * Оба меняют поведение алгоритма. 
  * **Template Method** основан на **НАСЛЕДОВАНИИ (Inheritance)**. Он меняет *части* алгоритма на этапе компиляции. Класс жестко привязан к базовому классу (Пицца всегда будет Рецептом).
  * **Strategy** основан на **КОМПОЗИЦИИ (Composition)**. Он меняет *весь* алгоритм целиком на этапе выполнения (в рантайме), передавая объект алгоритма внутрь.
